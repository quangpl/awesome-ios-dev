# Contributing to Awesome iOS Development

Thank you for helping make this list better. Please read these guidelines before submitting a pull request.

## What Belongs Here

This list focuses on **production-quality resources** for modern iOS development. An entry qualifies if it meets ALL of the following:

- **Real and verifiable**: the project, tool, or resource actually exists at the linked URL.
- **Actively maintained**: libraries must have had a commit, release, or meaningful activity within the last 18 months. Abandoned projects will be removed.
- **Widely adopted or genuinely exceptional**: prefer libraries used in thousands of projects, or newer libraries with clear technical merit and community traction.
- **iOS-relevant**: directly applicable to building iOS/iPadOS apps with Swift, SwiftUI, or UIKit.

## What Does Not Belong

- Personal projects with fewer than 100 GitHub stars and no production usage evidence.
- Tutorials, blog posts, or YouTube videos (unless listed in Communities/Newsletters/Podcasts and broadly recognized).
- Paid tools without a meaningful free tier, unless they are industry-standard (e.g., Reveal, Instruments).
- Objective-C-only libraries with no Swift support.
- Duplicate entries that already exist under a different category.

## Entry Format

```
- [Name](https://canonical-url) - One concise sentence describing what it does, ending with a period.
```

Rules for descriptions:
- Lead with what the library **does**, not what it **is** ("Adds X" or "Provides Y", not "A library that...").
- Keep it to one sentence and under 120 characters.
- No em dashes. Use commas, colons, or parentheses instead.
- No trailing adjectives like "great", "awesome", or "amazing".

## Submitting a Pull Request

1. Fork the repository and create a branch: `git checkout -b add-libraryname`.
2. Add your entry to the correct section in alphabetical order within that section.
3. Verify the URL resolves to the canonical source (GitHub repo or official docs).
4. Run a quick spell-check on your description.
5. Open a PR with the title: `Add [Name] to [Section]`.
6. In the PR body, include: a one-line justification (stars, production usage, what gap it fills).

## Updating or Removing Entries

- If a URL is broken or a project is abandoned, open a PR or file an issue with evidence.
- If a better canonical URL exists for an entry (e.g., moved from Bitbucket to GitHub), submit a correction PR.

## Quality Over Quantity

This list aims to be **curated**, not exhaustive. A smaller list of highly reliable entries is more valuable than a large list with noise. Reviewers will decline entries that do not clearly improve the list.

## Code of Conduct

Be respectful. Criticism of a library's technical merit is fine; personal attacks are not. All contributors are expected to maintain a constructive tone in issues and pull requests.
